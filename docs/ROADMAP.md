# OpenCollect PoC Roadmap

更新日期：2026-05-26

## 目标

这个文档用于追踪 OpenCollect PoC 的后续迭代优先级。判断标准是：这个 PoC 是否能证明用户愿意把小红书内容沉淀到 OpenCollect，并且之后真的会回来找、看、整理。

## 状态约定

- `Done`：已完成并验证。
- `Doing`：当前正在做。
- `Next`：下一阶段优先做。
- `Backlog`：已确认价值，但暂不做。
- `Blocked`：需要外部条件或方案确认。

## 当前已完成

- `Done` 单条小红书链接收藏解析。
- `Done` 收藏页瀑布流。
- `Done` 笔记详情展示。
- `Done` 多图左右切换。
- `Done` 视频播放代理。
- `Done` 详情右上角 `x` 收起。
- `Done` 编辑收藏内容。
- `Done` 删除收藏内容，并支持 toast 撤销。
- `Done` 一键清空收藏，并支持网页内确认和撤销。
- `Done` 清空后的完整空态 UI。
- `Done` 收藏卡片和详情页的平台来源标识。

## 优先级路线

| 优先级 | 方向 | 状态 | 目标 | 验收标准 |
| --- | --- | --- | --- | --- |
| P0 | Go 后端 + JSON Store + 收藏 API | Next | 从 `localStorage` 迁移到后端本地 JSON Store，建立稳定 API 边界 | 刷新/换浏览器后数据仍存在；前端收藏列表从 API 读取；旧 localStorage 数据可迁移 |
| P1 | OSS/S3 云同步 | Backlog | 把本地 JSON 数据文件同步到低成本对象存储 | 启动时可从云端拉取；写入后可上传；支持备份和同步失败提示 |
| P2 | 解析稳定性、去重、重新抓取 | Backlog | 让收藏失败原因清楚，重复收藏可识别，旧收藏可刷新元数据 | 重复链接提示已存在；失败原因分层；支持重新抓取并更新标题/媒体/统计 |
| P3 | 搜索、筛选、标签整理 | Backlog | 证明收藏系统的“找回来”价值 | 可按标题/正文/作者/标签搜索；可按平台/类型筛选；可编辑标签 |
| P4 | 小红书视觉和交互打磨 | Backlog | 让内容流和详情更接近真实小红书使用感 | 卡片比例基于媒体比例；管理操作更弱化；详情阅读体验更完整 |
| P5 | 批量链接导入和导出 | Backlog | 在不碰账号登录态的前提下提升导入效率 | 支持多链接粘贴导入；展示成功/失败/重复结果；支持 JSON 导出/导入 |
| P6 | 上线底座 | Backlog | 支撑真实域名和多用户使用 | 用户数据隔离；API 限流；日志；备份；基础账号系统 |
| Hold | 绑定个人账号自动导入点赞/收藏 | Blocked | 通过用户账号导入小红书点赞/收藏列表 | 暂不推进；缺少适合普通个人账号的官方授权 API，避免托管用户登录态 |

## P0 拆解：Go 后端 + JSON Store + 收藏 API

- [ ] `OC-P0-001` 确认 Go + Gin + JSON Store 作为后端主方案。
- [ ] `OC-P0-002` 定义 `data/collections.json` schema。
- [ ] `OC-P0-003` 初始化 Go 后端骨架和 Gin 路由。
- [ ] `OC-P0-004` 实现本地 JSON Store：加载、初始化、原子写、revision。
- [ ] `OC-P0-005` 新增收藏列表 API：`GET /api/collections`。
- [ ] `OC-P0-006` 新增创建/导入 API：`POST /api/collections/import-local`。
- [ ] `OC-P0-007` 新增编辑 API：`PATCH /api/collections/:id`。
- [ ] `OC-P0-008` 新增删除 API：`DELETE /api/collections/:id`。
- [ ] `OC-P0-009` 新增清空 API：`DELETE /api/collections`。
- [ ] `OC-P0-010` 前端从 API 读取和写入，不再以 `localStorage` 为主存储。
- [ ] `OC-P0-011` 首次打开时迁移旧 `localStorage` 数据到后端。
- [ ] `OC-P0-012` 验证编辑、删除、清空、撤销、平台角标在 API 存储下仍正常。

## 跟踪规则

1. 每次只把一个方向标记为 `Doing`。
2. 每次迭代开始前，从本文件选择 1 到 3 个任务 ID。
3. 每次迭代结束后，把完成项改成 `[x]`，并更新状态表。
4. 每次迭代结束后，在 `docs/ITERATION_LOG.md` 追加一条记录。
5. 如果发现任务不再成立，不删除，改成 `Blocked` 或在日志里说明原因。
